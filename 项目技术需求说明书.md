

# 项目技术需求说明书 (PRD & 架构设计)

**项目名称**：Obsidian Sync & Share (简称 OSS)

**项目目标**：开发一款类似 Fast Note Sync 的 Obsidian 插件及独立 Go 后端。提供多端静默同步（含 Markdown、附件及 `.obsidian` 配置）、版本冲突解决、支持多用户的私有化轻博客系统（带纯前端主题和自定义代码注入）、以及多人协作编辑功能。

**技术栈**：

- **后端**：Go + Gin (或 Echo) + GORM + robfig/cron
    
- **前端 (Obsidian 插件)**：TypeScript + Obsidian API + diff-match-patch (冲突对比)
    
- **主题引擎**：Go 原生 `html/template` (输出基础 Shell) + 纯前端静态主题 (CSS/JS) + 动态配置注入。
    

##  零、 核心架构原则 (AI 开发必读)

在进行任何代码编写前，请严格遵守以下核心决策：

1. **数据库选型**：默认使用 **SQLite**，但必须使用 **GORM** 编写，保证只需修改 DSN 即可无缝切换到 **PostgreSQL**。
    
2. **安全与加密**：采用 **“传输加密 (HTTPS) + 存储不加密 (服务器明文保存文件)”** 方案。绝对不使用端到端加密，因为服务器需要直接读取 Markdown 内容以渲染博客页面。
    
3. **同步机制**：采用基于时间戳的 **Last Write Wins (最后写入胜出)** 策略。通过对比客户端与服务端的 `mtime` 决定上传/下载，不记录 Git 式的 Diff 增量。（冲突检测的基准线问题见**决策 1**——严格 LWW 需在客户端维护 `.oss-sync-state.json` 基线才能判定 `conflict_detected`。）
    
4. **博客渲染与主题系统**：采用 **“纯前端静态主题 + 配置项驱动”**。Go 后端仅用 `html/template` 输出一个基础 HTML，并在其中注入解析好的 Markdown 内容、`ThemeConfig` (转为 JS 变量)，以及用户自定义的 `CustomHeader`/`CustomFooter`。页面美化完全交由外部静态 CSS/JS 处理。
    
5. **目录树保留**：默认保留 Obsidian 库的完整相对路径结构（如 `笔记/前端/React.md`），用户也可在插件端配置是否扁平化，默认开启。
    

## 一、 数据库设计 (GORM 模型)

请基于以下表结构在 Go 中生成 GORM 的 Models，并开启 AutoMigrate。

**1. User (用户表)**

- `ID` (uint, primary key)
    
- `Username` (string, unique, index)
    
- `PasswordHash` (string) // bcrypt 加密
    
- `Role` (string) // "admin" 或 "user"
    
- `StorageQuota` (int64) // 字节数限制，用于多用户管理
    

**2. UserSetting (用户配置表)**

- `UserID` (uint, unique index, 外键)
    
- `SyncInterval` (int) // 自动同步防抖时间(秒)，默认 300 (5分钟)
    
- `RecycleBinDays` (int) // 回收站保留天数，默认 30。0 代表不进回收站直接硬删除。
    
- `ThemeName` (string) // 博客静态主题文件夹名称，默认 "default"
    
- `ThemeConfig` (json) // 配合纯静态主题的配置 (JSON，包含是否开启评论、主题色等，由主题决定，后端透传)
    
- `CustomHeader` (text) // 自定义注入到 `<head>` 的 HTML/CSS/JS 代码
    
- `CustomFooter` (text) // 自定义注入到 `</body>` 前的 HTML/JS 代码
    
- `KeepDirectoryTree` (bool) // 是否保留目录树，默认 true
    

**3. File (文件元数据表)**

- `ID` (uint, primary key)
    
- `UserID` (uint, index)
    
- `Path` (string, index) // 库内相对路径 (如 `Notes/Go.md` 或 `.obsidian/app.json`)
    
- `Type` (string) // "markdown" 或 "attachment" 或 "config"
    
- `Hash` (string) // 文件内容 SHA256，用于快速比对文件是否实质改变
    
- `MTime` (int64) // 客户端最后修改时间戳 (基于 Last Write Wins 的核心凭据)
    
- `IsDeleted` (bool, default false) // 软删除标记 (移入回收站)
    
- `DeletedAt` (datetime) // 用于配合 Cron Job 定时清理回收站
    

**4. Share (分享记录表)**

- `ShareID` (string, primary key, uuid 或 6位短链)
    
- `UserID` (uint, index)
    
- `TargetPath` (string) // 目标路径，如 `Notes/Go.md` 或 `Notes/`
    
- `IsFolder` (bool) // 标识是否为整个文件夹分享
    
- `AllowCopy` (bool) // 是否允许前端一键复制 Markdown 源码
    
- `Views` (int) // 浏览量 (轻量级统计)
    
- `CreatedAt` (datetime)
    

**5. Collaboration (协作表)**

- `ID` (uint)
    
- `FileID` (uint, 外键, 关联具体的 Markdown 文件)
    
- `OwnerID` (uint) // 分享者
    
- `CollaboratorID` (uint) // 被邀请的协同者
    
- `Status` (string) // "pending" (待接受) / "accepted" (已接受)
    

## 二、 后端核心 API 与算法设计

### 1. 核心同步系统 (Last Write Wins & 配置同步)

**核心路由：`POST /api/sync/check`**

- **输入**：客户端提交本地文件元数据数组 `[{path, mtime, hash}]` (包含 `.md`, 附件, 以及 `.obsidian` 配置目录下的文件)。（收集时过黑名单，见**决策 6.1**；协议支持增量模式，见**决策 6.5**。）
    
- **处理逻辑**：
    
    - 客户端 `mtime` > 服务端 `mtime`：状态为 **upload_needed**。
        
    - 服务端 `mtime` > 客户端 `mtime`：状态为 **download_needed**。
        
    - **协作冲突判定**：如果两端 hash 不同，且服务端 `mtime` 更新（被协作者改过），且客户端本地也有新修改未上传，则触发 **conflict_detected**。（具体判定靠客户端基线，见**决策 1**；冲突弹窗用两方 Diff，见**决策 2**。）

> **响应须带 `server_time`**：check 响应顶层必须返回服务端当前时间戳，供客户端计算时钟偏移，见**决策 7.1**（LWW 抗时钟漂移的硬约束）。

- **文件传输路由**：

    - `POST /api/sync/upload`：接收文件流，保存至服务器物理磁盘 `data/{user_id}/{path}`。（必须 `io.Copy` 流式落盘，禁内存读取防 OOM，见**决策 7.3**；客户端并发上限见**决策 7.2**。）

    - `GET /api/sync/download`：下发文件流。（同样流式，见**决策 7.3**。）
        
    - `POST /api/sync/delete`：不直接删除磁盘文件，而是标记 `is_deleted = true`，记录 `deleted_at`（软删除机制）。
        

### 2. 轻博客、文件夹渲染与双向链接处理

**核心路由：`GET /p/:share_id`** (及通配符用于文件夹分享 `GET /p/:share_id/*subpath`)

- **基础 Shell 渲染逻辑**：
    
    1. 用 `html/template` 加载基础 HTML。
        
    2. 读取用户的 `ThemeName` 挂载对应静态资源 (`/themes/xxx/style.css`)。
        
    3. 注入 `ThemeConfig`、`CustomHeader` (安全转义后塞入 `<head>`) 和 `CustomFooter`。（Header/Footer 实际不转义、用 `template.HTML`，见**决策 4**；ThemeConfig 用 `template.JS`，见**决策 5**。）
        
    4. 如果是单篇分享，输出 Markdown 转换的 HTML；如果是文件夹分享，自动遍历该目录生成左侧目录树结构。
        
- **双向链接解析 (Backlink Parser)**：
    
    - 在 Go 后端使用 Markdown 解析库（如 `yuin/goldmark`）时，自定义正则或扩展拦截 `[[链接文字]]` 语法。（匹配规则见**决策 3**：在当前用户 shares 表全局匹配，仅已分享者渲染为链接。）
        
    - 判断规则：如果 `链接文字.md` 存在于当前分享的**文件夹**内，或者该文章被该用户独立分享过（在 `shares` 表有记录），则将其渲染为 `<a href="/p/{对应share_id}" target="_blank">链接文字</a>`。（同名歧义取最近更新，见**决策 3**。）
        
    - 如果未分享，渲染为 `<span class="unshared-link">链接文字(未分享)</span>`。
        

### 3. 定时任务清理系统 (Cron Jobs)

基于 `robfig/cron/v3`，每天凌晨执行以下两大任务：

1. **回收站清理**：执行 SQL 查找 `is_deleted == true` 且到期的记录 (`time.Now() - deleted_at > user_settings.RecycleBinDays`)。对查出的结果，先物理删除磁盘上的文件，然后从 `files` 表中硬删除记录。
    
2. **孤儿附件/图片清理**：
    
    - 扫描数据库中该用户所有正常的 Markdown 文件，正则提取 `![]()` 和 `[[]]` 中的附件路径，存入内存 Map。（正则须覆盖四类引用：标准MD、Obsidian双链、YAML Frontmatter、HTML `<img>`，见**决策 6.3**，否则误删。）
        
    - 遍历用户在服务器磁盘的附件目录，如果某文件路径不在 Map 中，并且其创建时间超过 24 小时（防止刚上传还没插入文档的图片被误删），则执行物理删除以释放空间。
        

## 三、 Obsidian 插件端 (TypeScript)

### 1. 全自动防抖同步机制

- 生命周期：使用 `this.app.vault.on('modify', ...)`、`'create'`、`'delete'` 监听文件库变动。（三个监听器内均需先过同步黑名单与 `.obsidian` 毒药文件过滤，见**决策 6.1、6.2**。）
    
- **Debounce (防抖) 队列**：用户在持续打字时不要触发请求。引入 `lodash` 的 `debounce` 函数，监听键盘输入，停顿 X 秒（读取用户设置的 `SyncInterval`，默认 300 秒/5分钟）后，将变动的文件加入待同步队列，并静默执行 `/api/sync/upload`。
    
- 同时提供一个 UI 按钮允许“立即强制同步”。
    

### 2. 协作编辑与冲突解决 (Diff UI)

- 同步检查 (`/api/sync/check`) 返回 `conflict_detected` 状态时，插件必须暂停当前文件的自动同步。
    
- 调用 `/api/sync/download?path=xxx` 临时将云端最新版本拉取到内存中（不覆盖本地文件）。
    
- 利用 Obsidian API 弹出一个 Modal 窗口。
    
- 在 Modal 中，调用 `diff-match-patch` 库，分屏或红绿底色高亮展示“我的修改”与“他人的修改”。（两方 Diff，不做三方 merge，见**决策 2**。）
    
- 提供三个操作按钮：
    
    1. **接受云端覆盖本地** (Accept Remote)
        
    2. **强制本地覆盖云端** (Force Push Local)
        
    3. **保留双方并产生副本** (Keep Both) -> 自动在本地当前目录生成 `原文件名_conflict_时间戳.md`，随后触发正常同步。
        

### 3. 分享与右键菜单

- 使用 `this.registerEvent(this.app.workspace.on('file-menu', ...))`，在文件树右键菜单添加 "分享至轻博客"。
    
- 弹窗设置项：
    
    1. 目标：当前单篇，还是整个文件夹？
        
    2. 是否允许访问者一键复制源码？
        
    3. 是否递归分享当前文章中包含的所有双向链接文章？（若勾选，插件自动解析文件内双链，向后端批量请求生成 `share_id`）。
        
- 后端返回 URL 后，调用 Obsidian API (如 `navigator.clipboard.writeText`) 将短链接自动写入剪贴板。

## 五、 开发决策与澄清 (AI 必读，优先级高于上文任何模糊处)

> 以下五条决策已由项目负责人拍板，**直接作为开发约束**。当上文任何描述与本节冲突时，**以本节为准**。每条均标注了它修正的上文章节，请在对应 Phase 实现时严格遵守。

### 决策 1：冲突检测的“基准线”问题（修正「零、3」与「二、1」）

**背景**：严格 LWW 下服务端只保留最新 mtime 版本，无法判断“客户端是否有未上传的本地修改”，因此原 `conflict_detected` 判定缺少基准线，逻辑不闭环。

**决策（方案A，行业标准）**：在 Obsidian 插件端维护本地基线文件 `.oss-sync-state.json`（隐藏，存于 vault 根或插件数据目录）。

- **字段结构**：`{ "<vault相对路径>": { "serverMTime": <上次成功同步时服务端返回的 mtime>, "hash": <上次同步时的 hash> } }`。
- **冲突判定逻辑（在 `POST /api/sync/check` 的客户端侧补全）**：
  - 若 `本地 mtime > 基线 serverMTime` 且 `服务端 mtime > 基线 serverMTime` → **两端都改过 → `conflict_detected`**。
  - 若仅 `本地 mtime > 基线` → `upload_needed`。
  - 若仅 `服务端 mtime > 基线` → `download_needed`。
  - 若 `本地 mtime == 基线 == 服务端 mtime` → `in_sync`。
- **基线更新时机**：仅在某文件 **成功 upload 或 download 完成后**，用服务端返回的最新 mtime/hash 覆盖该路径条目。失败、冲突未解决期间不得更新基线。
- **首次同步**：本地无基线条目时，视为 `download_needed`（拉取服务端 mtime 建立基线）；若服务端也无记录，视为 `upload_needed`（上传后建立基线）。

> ⚠️ 此决策不修改 File 表结构（不在服务端加 BaseMTime 字段），基线完全在客户端维护。服务端 `/api/sync/check` 的响应仍按「二、1」返回各文件的服务端 mtime 与 hash，由客户端用本地基线做冲突判定。

> **已知限制（方案A 固有取舍）**：基线是设备本地文件，**同一用户在多台设备间编辑时，冲突检测会退化**——第二台设备的 `.oss-sync-state.json` 没有第一台写入的服务端 mtime 基线。其后果：第二台设备仍能靠服务端返回的 mtime 正常 `download_needed`/`upload_needed`，但若两台设备在同步间隔内**都**改了同一文件，第二台上传时服务端会按 LWW 直接覆盖第一台的修改，**不会触发 `conflict_detected`**（因为第二台没有“第一台改过”的基线参照）。若后期发现多设备冲突漏检严重，再考虑在 File 表加服务端 `LastSyncedMTime` 字段做服务端侧兜底判定。

### 决策 2：冲突 Diff 采用两方对比（修正「三、2」）

**决策（方案A，开发轻量）**：冲突弹窗只做**两方 Diff（本地当前文件 vs 云端最新文件）**，不做三方 merge。

- 使用 `diff-match-patch` 对本地文本与云端文本直接做两方 diff，红绿底色高亮差异。
- 用户操作仅限「二、1」描述的三选一：
  1. **Accept Remote**：用云端覆盖本地 → 客户端 download 后更新基线。
  2. **Force Push Local**：用本地覆盖云端 → 客户端 upload 后更新基线。
  3. **Keep Both**：在本地当前目录生成 `原文件名_conflict_<时间戳>.md`（本地修改内容另存副本），原文件保持云端最新（download），随后对新副本触发正常 upload。由用户事后手动合并。
- **不提供**逐行 cherry-pick / 三方合并 UI（超出本期范围）。

### 决策 3：双向链接采用“全局匹配，仅已分享”规则（修正「二、2」）

**决策（方案A，安全可控）**：渲染 `[[链接文字]]` 时，不做当前文件夹内的局部匹配，而是在**当前用户的 `shares` 表（含单篇与文件夹分享）中全局查找**文件名为 `链接文字.md` 的文章。

- 查到唯一 → 渲染为 `<a href="/p/{对应share_id}" target="_blank">链接文字</a>`。
- 查到多个同名 → 取 `CreatedAt` 最近更新的那条 share_id（同名歧义以最新分享为准）。
- 未查到（未分享）→ 渲染为 `<span class="unshared-link">链接文字(未分享)</span>`。
- 文件夹分享下，若 `[[链接文字]]` 指向的文章落在该文件夹范围内但未被单独分享，仍可命中（因为文件夹分享覆盖其下所有文件）——此规则在「文件夹分享遍历」时由后端预先把文件夹内所有文件视为“可被双链命中”，合并进全局查找集合。

> 实现提示：建议在 `/p/:share_id` 渲染前，一次性加载该用户的 share 集合与（若是文件夹分享）该文件夹下的全量文件路径，构建 `文件名 → share_id` 的内存索引，供 Goldmark 扩展 O(1) 查询，避免逐链接查库。

### 决策 4：CustomHeader/Footer 信任边界内不转义（修正「二、2」）

**决策（方案A）**：`CustomHeader`、`CustomFooter` 为**信任边界内功能**，使用 `template.HTML` 原样输出，**不做 HTML 转义**。

- 理由：私有化部署，用户在后台注入统计代码（百度统计、Google Analytics）等是合理诉求，转义会让其失效。
- 安全模型声明：用户自己注入的恶意脚本只影响其本人分享出去的页面，由用户自行承担风险。
- **仍需防御**：`ThemeConfig`（JSON，来自后台结构化配置）走决策5的 `template.JS` 注入；普通用户输入的博客正文 Markdown 走 Goldmark 正常渲染（goldmark 默认对原始 HTML 做安全限制，必要时启用 `html.Renderer` 的白名单或禁用 raw HTML）。
- 权限：仅 `Role == "admin"` 可设置 `CustomHeader/Footer`（在 API 层校验）。

### 决策 5：ThemeConfig 注入使用 template.JS

**决策**：在 Go 模板中注入 `ThemeConfig`（JSON）为 JS 变量时，使用 `template.JS` 类型断言，避免 `html/template` 默认 HTML 转义破坏 JSON 结构。

```go
// 正确写法
tmpl.Execute(w, map[string]interface{}{
    "ThemeConfig": template.JS(themeConfigJSONBytes), // 不被转义
})
```

模板侧：

```html
<script>window.__THEME_CONFIG__ = {{.ThemeConfig}};</script>
```

> 同样原则适用于任何注入到 `<script>` 内的 JSON 变量。注入到 HTML 正文位置的字符串仍用默认转义。

### 决策 6：落地隐患与边界情况兜底（修正「二、1」「二、3」「三、1」）

以下为开发中极易翻车的边界情况，统一在此固化对策，避免开发到 Phase 3/6 时才暴露。

#### 6.1 同步绝对黑名单（防止状态文件循环同步）

**坑**：若插件把 `.oss-sync-state.json` 自身当成普通文件上传，Mac 的基线会覆盖 PC 的基线，冲突检测逻辑瞬间崩溃。

**对策**：在 Obsidian 插件源码中**硬编码绝对黑名单**，执行 `/api/sync/check` 收集本地文件树时直接剔除（用户不可配置关闭）：

- `.oss-sync-state.json`（本项目自身基线）
- `.git/`、`.DS_Store`、`Thumbs.db`（系统/版本控制噪声）
- `.trash/`（Obsidian 回收站，已有自己的回收站机制）

> 黑名单在客户端生效即可，服务端无需感知。`'create'/'modify'/'delete'` 三个 vault 监听器内都需先过黑名单过滤再入队。

#### 6.2 `.obsidian` 目录的毒药文件过滤

**坑**：同步 `.obsidian/` 会把 `workspace.json`、`workspace-mobile.json` 带过去——这俩记录面板布局/窗口大小/光标位置，PC↔手机互相同步会互相把界面拉扯成对方布局，体验极差。

**对策**：同步 `.obsidian/` 时**默认过滤**以下文件（可在插件设置中单独开关，默认关）：

- `workspace.json`、`workspace-mobile.json`
- `.obsidian/cache`、`.obsidian/plugins/*/data.json`（插件本地状态，跨端意义不大且易冲突）

> 保留同步：`app.json`、`appearance.json`、`community-plugins.json`、`core-plugins.json`、`graph.json`、`hotkeys.json`、主题与插件本体——这些是真正想跨端一致的配置。

#### 6.3 孤儿附件正则必须覆盖全场景（防误删）

**坑**：Phase 6 孤儿清理若正则只认 `![[image.png]]` 和 `![](image.png)`，会漏掉 YAML Frontmatter 封面图、HTML `<img>` 标签等引用，导致凌晨 Cron 把仍被引用的附件物理删除。

**对策**：后端 Phase 6 提取附件的正则必须**至少覆盖以下四类**：

1. **标准 Markdown 图片**：`!\[.*?\]\((.+?)\)`，捕获分组1为路径。
2. **Obsidian 双链图片**：`!\[\[(.+?)(?:\|.+?)?\]\]`，捕获分组1，去掉 `|别名/尺寸` 部分。
3. **YAML Frontmatter 区块**内的图片字段（如 `cover:`、`image:`、`banner:`），需先切出文件首部 `---`...`---` 区块再按 `key: value` 提取其中的相对路径。
4. **HTML `<img>` 标签**：`<img[^>]+src=["'](.+?)["']`，捕获分组1。

> 路径归一化：捕获到的路径统一去掉 `./` 前缀、解析 `../`，与 File 表 `Path` 字段对齐后再比对内存 Map。24 小时新文件保护期（防刚上传未插入文档）保留不变。

#### 6.4 文件重命名/移动导致分享链接失效（接受妥协）

**坑**：Obsidian 本地重命名 A.md→B.md，本地双链自动更新；但同步系统视为“Delete A + Create B”，A 进回收站后，绑在 A 路径上的 `share_id` 渲染 404。

**对策（显式妥协，不做过重的路径追踪）**：

- 重命名已分享文章后，旧 `share_id` 失效。用户需在右键菜单点“**更新分享链接**”为新路径重新生成 `share_id`。
- 在插件分享设置 Modal 与后端博客首页文案中**明确告知用户**：“重命名/移动已分享的文章会导致原链接失效，需重新分享。”
- 不实现自动路径迁移（成本高、收益低）。

> 后端 `/p/:share_id` 渲染时，若 `TargetPath` 对应文件已 `IsDeleted=true` 或物理不存在，返回友好的“此内容已被作者移除或重命名”页面，而非裸 404。

#### 6.5 首次全量同步 Payload 优化（客户端增量 check）

**坑**：`POST /api/sync/check` 每次发全量 `{path, mtime, hash}` 数组，万级文件 Vault 的 JSON 可达数 MB，移动端网络浪费。

**对策（非阻塞优化，可后置）**：插件端做**增量 check 缓存**——

- 本地维护“自上次 check 以来 mtime 未变化的文件”集合，常规 check 只提交**本端 mtime 有变动的文件**；对未变动文件，假设服务端也未变（除非用户点“**立即强制全量校验**”按钮走全量）。
- 强制全量校验按钮每次提交全量数组，用于纠正可能的漂移（如服务端被其他设备改过但本端文件未动）。
- 协议上 `/api/sync/check` 请求体增加一个 `mode: "incremental" | "full"` 字段；`incremental` 模式下，服务端只对提交的文件返回比对结果，未提交文件返回 `assume_in_sync` 占位（客户端不处理）。

> 此优化可放至 Phase 3 联调跑通后、性能阶段再实现，不阻塞 MVP。但协议字段（`mode`）应在 Phase 2 设计 API 时就预留，避免后期破坏性改协议。

### 决策 7：上线即崩的性能与边界防范（修正「二、1」「二、2」「三、1」）

以下三个问题若不在开发期防范，上线第一周必崩。均需在对应 Phase 实现时内建，不可后置。

#### 7.1 时钟漂移对齐（LWW 的天敌，必做）

**坑**：LWW 完全依赖 mtime。用户主板电池没电或手动调快系统时间，会使本地修改“来自未来”，其他设备永远无法覆盖该版本，同步彻底混乱。

**对策**：

- `/api/sync/check` 的响应 JSON 顶层增加 `server_time`（服务端当前 Unix 毫秒时间戳）。
- 插件端每次 check 收到后，计算 `time_offset = server_time - 本地当前时间戳`，缓存于内存（每次 check 刷新）。
- 插件端上传文件时，提交的 `mtime` = `本地真实 mtime + time_offset`，**强行以服务器时间为基准**。
- 若 `|time_offset| > 300秒`（5分钟），插件在状态栏提示“检测到本地时钟与服务端偏差过大，已自动校正；建议同步系统时间”。
- 服务端 download 时返回的 mtime 已是服务端时间，客户端直接存入基线，无需再加 offset。

> 此决策直接影响 Phase 2 的 `/api/sync/check` 响应结构与 Phase 3 的上传逻辑，必须在 Phase 2 设计 API 时就把 `server_time` 字段加进响应。

#### 7.2 并发上传风暴控制（防 FD 耗尽，必做）

**坑**：首次全量同步 2000 文件时，若插件 `files.map(f => fetch('/upload', f))` 瞬间发起 2000 并发请求，客户端卡死、服务端 `too many open files` 宕机。

**对策**：

- 插件端实现**并发控制队列**，限制同时 in-flight 的上传/下载请求上限（默认 `MAX_CONCURRENCY = 6`，可在设置中调，范围 1–10）。
- 推荐用 `p-limit` 或自实现信号量：`const limit = pLimit(6); await Promise.all(files.map(f => limit(() => upload(f))))`。
- 单请求超时设 60s，失败按指数退避重试最多 3 次，重试仍失败的文件标记为 `sync_failed` 入“失败重试队列”，不阻塞其他文件。
- 服务端 Gin 层额外加一道保险：设置 `MaxMultipartMemory` 合理值与中间件级的并发连接上限，作为客户端限流失效时的兜底。

#### 7.3 大文件流式上传（防 OOM，必做）

**坑**：用户存几十/上百 MB 的 PDF/MP4，若 Go 用 `c.ShouldBind` 一次性读入内存再落盘，几个并发大文件即触发 OOM 被杀。

**对策**：

- Go upload 处理**绝对禁止**把整个文件读入内存，必须用 `io.Copy(file, requestBody)` / ` multipart.File` 流式写入磁盘。
- `MaxMultipartMemory` 设为较小值（如 8MB）——仅对内存中缓存的 multipart 表单部分生效，超出部分自动落临时文件，不影响流式拷贝。
- 若前置 Nginx，`client_max_body_size` 调至与单文件上限一致（如 100M）。
- 单文件大小上限：MVP 阶段设 `MAX_FILE_SIZE = 100MB`（可配置），超出直接拒绝并提示用户。此上限同时作为 Phase 2 upload 校验逻辑的一部分。
- download 同样流式：`io.Copy(w, file)` + 设置 `Content-Length`/`Content-Disposition`，禁止缓冲整文件。

> 此决策影响 Phase 2 的 upload/download 实现，是硬约束，不可用“先跑通再说”的内存读取写法偷懒。

### 决策 8：CI/CD 与多环境部署（指导性，具体平台待定）

**原则**：Go + Obsidian 插件架构不引入重运维（不上 K8s）。目标是自动化、低运维。

#### 8.1 Go 后端：Docker + CI 流水线

- **多环境**：不搞复杂预发环境，用两套配置 `config.dev.yaml`（本地 SQLite）与 `config.prod.yaml`（生产），或基于 `.env` 注入。配置加载逻辑需在 Phase 1 脚手架就支持“按环境变量 `OSS_ENV` 切换配置文件”。
- **流水线**（具体 CI 平台见下方待定项）：push → `go test` → 编译静态二进制 → 打包极小 Docker 镜像（< 20MB，用多阶段构建 + `alpine`/`scratch`）→ 推镜像仓库。
- **部署**：生产服务器 `docker-compose.yml` 挂载 `./data` 目录持久化（SQLite 文件 + Markdown 全部在 data 下，容器删除数据不丢）。可选 Watchtower 做镜像更新自动重启。
- **数据备份**：`./data` 目录需配合宿主机定时任务（cron + `sqlite3 .backup` + 文件打包）做每日备份，本决策仅声明“必须有备份”，具体备份策略由运维侧落实。

#### 8.2 Obsidian 插件：Tag 触发 Release

- **开发**：本地 `esbuild --watch` 热编译输出到测试 Vault 的 `.obsidian/plugins/oss/`，实时看效果。
- **CI**：打 Git Tag（如 `v1.0.0`）推送 → CI 执行 `npm run build` → 自动打包 `main.js`、`styles.css`、`manifest.json` → 创建 GitHub Release。
- **分发**：内测用 Obsidian **BRAT 插件**输入仓库地址一键安装/热更新；稳定后提交官方插件市场。
